## 1. Authentication, Authorization & Access Control

### 🟥 Bugs
- **A1 — IDOR `GET /api/tracks/:id`** — `server/routes.ts:4737`. Returns the track (incl. `audioUrl`) for any id; the sibling PATCH/DELETE check `existing.userId !== req.userId` but this GET was missed. **Fix:** `if (track.userId !== req.userId && !req.isOwner) return sendError(res, 403)`.
- **A2 — IDOR `GET /api/tracks/project/:projectId`** — `server/routes.ts:4714`. `getProjectTracks(projectId)` with no check the project belongs to the caller. **Fix:** verify `project.userId === req.userId` first.
- **A3 — Owner locked out of pro features** — `server/middleware/auth.ts:116`. `requireSubscription` does `storage.getUser('owner-user')`, but no DB row has that id, so the owner gets **402** on every gated route. `requireAuth()` special-cases `owner-user`; this middleware doesn't. **Fix:** `if (req.isOwner || req.userId === 'owner-user') return next();` at the top.

### 🟧 Risks
- **A4 — `/owner-login` session fixation** — `server/routes/auth.ts:160`. Sets `req.session.userId` without `regenerate()`/`save()` (unlike `/login` at `:111`) — fixation against the highest-privilege account. **Fix:** mirror `/login`.
- **A5 — Two inconsistent "owner" identities** — `keys.ts:37` (DB-user UUID via session) vs `middleware/auth.ts:24` (`'owner-user'` via `x-owner-key` header). The split is the root cause of A3 and unpredictable owner rights. `keys.ts:37` also uses plain `===` on the owner key (not constant-time, unlike the middleware's `safeEqualString`). **Fix:** unify on one owner identity; use `safeEqualString`.
- **A6 — Dev secret fallbacks** — `server/lib/jwt.ts:12` and `server/index.ts:296` fall back to literal `"dev_..._change_me"` secrets. Prod boot is guarded by `validateEnv()`/jwt prod-exit, so contained — but anyone running without `NODE_ENV=production` on a reachable network forges valid JWTs with a public default. **Fix:** fail fast whenever the literal default is in use, regardless of NODE_ENV.
- **A7 — CORS wildcard + credentials** — `server/index.ts:191`. If `CORS_ALLOWED_ORIGINS` ever contains `*`, the code reflects the caller Origin **and** sets `Allow-Credentials: true` — a credentialed-CORS bypass. **Fix:** refuse to combine wildcard with credentials.
- **A8 — WebEar relay key in query string + open CORS + unauth `/blob`** — `server/routes/webearRelay.ts:251,266,298`. Accepts `wbr_` keys from `req.query.key` (leaks to logs/Referer), sets `Allow-Origin: *` everywhere, and `POST /blob/:captureId` has **no auth** (anyone guessing an 8-byte captureId overwrites a pending capture). **A9** — `/api/webear/analyze-app/:captureId` (`:316`) is unauthenticated and reads any capture by id.

### 🟨 Smells
- **A10** — Guest user created with blank password (`guestUser.ts:28`); keep strictly read-only.
- **A11** — `/api/keys/activate` (`keys.ts:17`) is unauthenticated, enumerable, and creates a session as the matched user → brute-forceable account hijack. Add a strict dedicated limiter; ensure high-entropy keys.
- **A12** — `authLimiter` trusts the first `x-forwarded-for` element while `validate.xForwardedForHeader:false`; verify `trust proxy` + a trusted XFF source so the 100/15-min cap can't be evaded by header rotation.

**Verified OK:** `/login` regenerates session + bcrypt cost 10; session cookie `httpOnly/secure(prod)/sameSite` sound; owner-key header compare constant-time; dev auto-login disabled in prod; most `/api/user/*` and song mutations derive identity from `req.userId`.

## 2. Money Path (Stripe / credits) — prior fixes re-verified ✅

All four 2026-04-30 fixes are **present and correct**: single Stripe API-version source (`services/stripe.ts:20`), webhook signature verify on raw body mounted before `express.json` (`stripe.ts:89`, `index.ts:248`), atomic claim-execute-release idempotency (`storage.ts:2093`, `stripe.ts:110`), transactional `grantCreditsAtomic` (`storage.ts:2058`), server-side price/amount/currency re-validation (`routes/credits.ts:294`), and refund/dispute clawback (`stripe.ts:303`).

### Remaining findings
- **M1 🟥 — Unvalidated `packageKey` from metadata** — `services/stripe.ts:162`. Cast `as keyof typeof CREDIT_PACKAGES` with only a truthy check; a valid-but-wrong key grants the wrong package, a garbage key throws. **Fix:** `if (!(packageKey in CREDIT_PACKAGES)) throw ...` before `purchaseCredits`.
- **M4 🟧 — Spend path not transactional** — `services/credits.ts:178`. `deductCredits` does `atomicDeductCredits` then a **separate** `logCreditTransaction`; a crash between them spends credits with no ledger row. The grant path was made transactional but the deduct path wasn't. **Fix:** wrap deduct + ledger insert in one `db.transaction`.
- **M3 🟧 — Clawback bounded to 500 unindexed rows** — `services/stripe.ts:323`. `getCreditTransactions(user.id, 500, 0)` + JS `.find` on `paymentIntentId`; a heavy user with >500 newer txns since purchase **misses the original** → no clawback on a real refund/chargeback. **Fix:** indexed `WHERE metadata->>'paymentIntentId' = $1 AND amount > 0` (the query `hasProcessedPaymentIntent` already uses).
- **M2 🟧** — `getCreditService` singleton binds the first `storage` passed (`credits.ts:397`); latent footgun if two IStorage instances ever exist. Key by WeakMap or drop the singleton.
- **M6 🟨** — `billing.ts:20` validates `tier ∈ {basic,pro}` but `createCheckoutSession` ignores it and always uses the pro `PRICE_ID` — `basic` silently checks out as pro.
- **M5 🟨** — owner balance returns `Number.MAX_SAFE_INTEGER` (`routes/credits.ts:65`); magic value leaks into ledger `balanceBefore` arithmetic.
- **M7 🟨** — `invoice.paid`/`payment_failed` mutate subscription status **before** the idempotency claim (`stripe.ts:239`); idempotent today, but any future side-effect here double-fires on retries.

## 3. Data Layer, Social & Validation

### 🟥 Bugs
- **D1 — `getSocialFeed` filters after limiting** — `storage.ts:2534`. Pulls the 50 newest posts **globally** then JS-filters to self+followees, so a user can see almost none of their own/followees' posts. **Fix:** push the filter into SQL (`inArray(socialPosts.userId, [userId, ...followingIds])`) then `.limit(50)`.
- **D2 — Raw `sql\`... IN ${array}\`` won't expand** — `storage.ts:2494,2502`. postgres-js binds the JS array as a single param, not an `IN (...)` list. Values are DB-sourced UUIDs today (not a live injection) but it's a fragile pattern that becomes injectable if a user-tainted value ever flows through. **Fix:** `inArray(users.id, followerIds)`.

### 🟧 Risks
- **D5 — Shared guest identity = cross-IDOR** — `songs.ts:171-204,287-304`. Unauthenticated `/upload`, list, delete, and `/:id/public` fall back to one shared `getGuestUserId()`, so any anonymous user can list/delete/toggle-public **any other guest's** songs. Authenticated paths correctly check ownership; the guest collision is the hole.
- **D3 — Project-share IDOR** — `social.ts:111` → `storage.ts:2515`. `POST /api/social/share-project` never checks the caller owns `projectId`; any user can "share" anyone's private project (granting view/edit/admin). **Fix:** assert `project.userId === req.userId` first.
- **D6 — Unbounded pagination (DoS)** — `blog.ts:12` (`?limit=10000000`), `social.ts:593` discover, `storage.ts:2650` `getUserConversations` (loads all messages, groups in JS). Clamp with `Math.min(parsed, 100)`.
- **D4** — `POST /api/social/chat/read/:conversationId` (`social.ts:554`) takes an attacker-controlled conversationId unvalidated against membership (storage filters by `recipientId` so impact is low).
- **D7 — N+1 enrichment** — `getSocialFeed`/`getPublicOrganismFeed`/`getUserConversations`/`getUserPendingInvites` `await getUser()` inside a `for` loop (up to 51 queries for a 50-row feed). Batch with one `inArray` fetch.

### 🟨 Smells
- **D8 — Stored-XSS exposure** — blog `content`/`title` (`blog.ts:79`), social post `content` (`social.ts:339`), chat `content`, profile `bio`/`displayName`, organism-session `caption` are stored raw with no server-side sanitization. Safety rests entirely on the React client escaping — and the blog body is rendered as HTML (see C1). Sanitize on write or render.
- **D9 — Schema hardening** (`shared/schema.ts`): `userFollows` (`:364`) lacks a unique `(followerId, followingId)` constraint and indexes (racy duplicate follows); `chatMessages` (`:409`), `socialPosts` (`:373`) lack indexes on their hot query columns (full scans); `projectShares` (`:321`) lacks unique + `onDelete`; `collabInvites` (`:423`) `fromUserId/toUserId/projectId` have **no `.references()`** (no referential integrity).
- **D10** — `share-organism-session` (`social.ts:315`) `JSON.parse` of unauthenticated input with no size cap; wrap in try/catch + length limits.

**Verified OK:** blog PUT/DELETE and authenticated song DELETE/public-toggle check ownership; collab-invite respond checks `toUserId`; blog/profile storage use parameterized Drizzle.

## 4. React Client

### 🟥 Bugs
- **C1 — Stored XSS sink** — `client/src/pages/blog/[slug].tsx:190`: `dangerouslySetInnerHTML={{ __html: post.content }}` on raw API HTML, zero sanitization. **Fix:** `DOMPurify.sanitize(...)` or a safe Markdown renderer.
- **C2 — JWT in localStorage** — `login.tsx:62`, `signup.tsx:66`, read in `lib/queryClient.ts:190,234`. JS-readable token; any XSS (C1) steals the session. **Fix:** httpOnly Secure SameSite cookie + `credentials:'include'` (already used elsewhere); drop client token storage.

### 🟧 Risks
- **C3 — JWT in SSE URL** — `hooks/use-voice-convert.ts:113`: `?token=${token}` leaks the JWT to access logs/history/Referer. **Fix:** authenticate the stream via cookie or a short-lived stream ticket.
- **C4 — Client-shipped key branch** — `lib/audioDebugBridge.ts:132` reads `import.meta.env.VITE_WEBEAR_API_KEY` (inlined into the public bundle if ever set). A safe auth-gated `/api/webear-keys/reveal` fallback already exists. **Fix:** delete the env branch. (`VITE_STRIPE_PUBLIC_KEY`/`VITE_GA_MEASUREMENT_ID` are public-by-design — fine.)

### 🟨 Smells / verified OK
- **C5** — `VulnerabilityScanner.tsx:42` hardcodes a fake `"sk-1234..."` — it's intentional demo "vulnerable code" sample, harmless but trips secret scanners.
- **Verified OK:** no `new AudioContext()` anywhere (CLAUDE.md rule upheld); no `eval`/`new Function`/`document.write` in app code; no insecure `window` `message` listeners; spot-checked `setInterval`/listener cleanup is clean.

## 6. AI Routes, File/Object Storage, SSRF & Command Injection

### 🟥 Bugs
- **F1 — SSRF in `extract-patterns`** — `server/routes/audio.ts:1401` (route `:1384`). `const { audioUrl } = req.body; await fetch(audioUrl)` with **no** `isAllowedUrl()` guard (unlike `songs.ts:1257`), then writes the body to disk. An authenticated user can hit `http://127.0.0.1:1106/...` (Replit sidecar that mints GCS creds), `http://169.254.169.254/...` (cloud metadata), or internal services. **Fix:** run `audioUrl` through `isAllowedUrl()` (mirror `songs.ts:1257-1262`).
- **F2 — Unauthenticated internal uploads** — `server/routes.ts:3980` (PUT) and `:4010` (GET). Neither has `requireAuth()`. Traversal is mitigated by `sanitizeObjectKey`, but any anonymous caller can **write up to 100 MB** to the objects dir (storage-fill / hosting abuse) and **read any object by key** (only protected by UUID unguessability, not authz). **Fix:** add `requireAuth()` + ownership checks; constrain PUT to the caller's key prefix; rate-limit.

### 🟧 Risks
- **F3 — SSRF allow-list bypassable** — `server/utils/security.ts:223` `isAllowedUrl()`. The single SSRF chokepoint blocks by **string-matching the hostname** and never resolves DNS → DNS-rebinding passes; also misses decimal/octal/hex IPs (`http://2130706433/`), IPv4-mapped IPv6 (`::ffff:127.0.0.1`), `[::]`. (Metadata `169.254.x` is caught.) **Fix:** resolve the host and validate the **resolved IP** against private/link-local ranges, then pin that IP for the fetch.
- **F4 — Unauth + uncredited ffmpeg decode (cost/CPU DoS)** — `server/routes/webearRelay.ts:298,316` (same as A8/A9). Anyone can POST a 50 MB blob under any `captureId` and trigger `ffmpeg` decode, no auth/credit/limit, CORS `*`. **Fix:** require API key, scope `captureId` to the session, rate-limit.
- **F5 — Vulnerability scanner has no rate limit / credit gate** — `server/routes/vulnerability.ts:18`. `POST /scan` is auth-gated but each call makes a GPT-4 completion (`max_tokens: 2000`) on up to 50k chars with **no limiter and no `requireCredits`** — an authenticated user loops it for unbounded AI spend. **Fix:** add `aiGenerationLimiter` and/or `requireCredits()`.

### 🟨 Smells
- **F6** — `vulnerability.ts:266` interpolates raw user code into the GPT-4 prompt; output is shape-validated and never drives a privileged action, so prompt injection only corrupts the scan result. Low impact.
- **F7** — `/api/ai/next-section` (`ai.ts:58`) is intentionally public; it runs a local engine (no external AI cost), but add an IP rate limit.

**Verified OK:** all `spawn('ffmpeg', [...])` calls use fixed argv arrays with `pipe:0/1` — **no command injection**; `objectStorage`/`sanitizePath`/`sanitizeObjectKey`/`validateFilename` properly confine local file ops (no traversal found); `voiceConvert` `sourceUrl` is never server-fetched (worker only accepts local paths); upstream API keys are never returned to the client or logged; multer/`express.raw` upload limits are set.

## Recommended remediation order

1. **Unauth / IDOR / SSRF** (F2, F1, A1, A2, D5, D3, A3, A4) — direct data exposure, privilege, and server-side-request bugs; mostly small fixes.
2. **XSS + token storage** (C1 + C2 together) — they compound into session theft.
3. **Money correctness** (M1, M4) — package validation + transactional spend.
4. **SSRF hardening + AI cost-DoS** (F3, F4, F5).
5. **Feed correctness + raw-SQL `IN`** (D1, D2).
6. **DoS / perf** (D6, D7), **deps** (`npm audit fix`).
7. **Schema hardening + repo hygiene** (D9, the committed `.exe` / audio assets).
