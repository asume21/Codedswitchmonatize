## TL;DR — top findings

1. 🟥 **Stripe API version mismatch** in `server/services/stripe.ts:20` vs `:41`. The new exported constant `STRIPE_API_VERSION = "2025-08-27.basil"` is never read — the constructor hardcodes `"2025-01-27.acacia"`. The comment claims this is the "single source of truth" — it isn't.
2. 🟥 **`purchaseCredits` self-cancels via the very dedup row the webhook just inserted.** `tryClaimStripeEvent` writes a row with `paymentIntentId`; `purchaseCredits` then calls `hasProcessedPaymentIntent(paymentIntentId)` which finds *that same row* and returns `null` without granting credits. **First-delivery purchases will silently drop on the live DB.** The unit test passes only because the storage mock hardcodes `hasProcessedPaymentIntent → false`.
3. 🟨 **`TransportContext.play()` is now async-wrapped** (`void (async () => …)()`), opening a stop-during-resume race window where a fast `stop()` gets reverted by the in-flight `play()` after `await resumeAudioContext()`. UX-visible only ("Stop didn't stop for 50ms"); no data corruption because Tone.Transport.start is idempotent on already-running state.
4. 🟧 **`AstutelyCoreContext` useEffect dep changed from `[]` to `[organismIsRunning]`** — the bridge connect/disconnect now fires on every play/stop cycle. State subscribers will see brief gaps and the rAF batcher restarts cold each time.
5. 🟨 **`AstutelyCoreContext.tsx:236, :465` carry literal `\"` escape leakage** in source (`\"replace\"`, `console.log(\`… \"${options.style}\" …\`)`). Compiles, but it's an obvious sed/heredoc artifact.

The big architectural moves (singleton AudioContext, ESLint invariant guard with cleanup ratchet, processed-events table, atomic credit grants, structured logger, env validation, integration test for webhooks) are **genuinely good work**. The bugs are all in the seams between those moves.

## Tier 1 — Money paths and runtime singletons

### `server/services/stripe.ts`

🟥 **API version regression.** Lines 20 and 41 contradict each other in the same file:
```ts
export const STRIPE_API_VERSION = "2025-08-27.basil" as const;  // line 20
…
return new Stripe(secretKey, { apiVersion: "2025-01-27.acacia" });  // line 41
```
The comment above the constant says "Single source of truth … an upgrade to one would silently leave the other on a stale version." The refactor introduced *exactly that bug*. Fix: `apiVersion: STRIPE_API_VERSION`. The webhook test at `server/api/__tests__/webhook.test.ts:49` uses `2025-08-27.basil`, so the test gives false confidence — it tests the constant, not the code.

**Verified empirically against installed SDK.** `package.json` pins `stripe: ^18.4.0`; installed version is 18.5.0. The SDK's `node_modules/stripe/types/lib.d.ts:30` declares `LatestApiVersion = '2025-08-27.basil'`, and `WebhookEndpointsResource.d.ts:150-158` lists both `2025-01-27.acacia` and `2025-08-27.basil` as accepted union members — so neither value is a TS-error workaround. The constant is correct (matches SDK's pinned latest); the constructor literal is a stale leftover from before this refactor. Net direction: the constructor is downgraded relative to the SDK's intent, not vice versa.

🟥 **Idempotency self-cancel for `checkout.session.completed`.** Walk-through:
- `claimEvent({ userId, paymentIntentId })` inserts a row into `processed_stripe_events` with `payment_intent_id = pi_xxx`.
- Then `creditService.purchaseCredits(userId, packageKey, paymentIntentId)` runs.
- That method (`server/services/credits.ts:312`) calls `storage.hasProcessedPaymentIntent(pi_xxx)`.
- `hasProcessedPaymentIntent` runs `SELECT * FROM processed_stripe_events WHERE payment_intent_id = ? LIMIT 1` (`server/storage.ts:1990`) — and finds the row `claimEvent` just inserted.
- Returns `true` → `purchaseCredits` returns `null` → `addCredits` never runs → user gets no credits.

The schema comment (`shared/schema.ts:78`) hints at the original intent — `// for cross-checks against legacy rows`. The dedup at the service layer was meant to catch *legacy* paymentIntents (from before the table existed), not the row the webhook just wrote. Three possible fixes, in **order of safety**:

1. **(Recommended) Make `hasProcessedPaymentIntent` mean what its name implies — "did we ever successfully grant credits for this PI?"** Join against `creditTransactions WHERE metadata->>'paymentIntentId' = $1 AND amount > 0` (or check the original purchase row). This preserves the service-layer admin-replay safety the comment cites and the legacy cross-check the schema comment cites, while no longer racing with the webhook claim.
2. Remove the service-layer dedup entirely — the webhook layer's claim is sufficient when `event.id` is the PK. **Sacrifices admin-replay safety** if `purchaseCredits` is ever invoked outside the webhook path.
3. Drop `paymentIntentId` from the row inserted by `claimEvent` for `checkout.session.completed` (rely on `eventId` PK alone). **Sacrifices the legacy-row cross-check capability** the schema names.

The unit test at `server/api/__tests__/webhook.test.ts:71` masks this: the mock `hasProcessedPaymentIntent: vi.fn().mockResolvedValue(false)` always returns false, which doesn't match the real DB behavior after `tryClaimStripeEvent` has written the row. Add a more faithful in-memory storage in tests, or assert that `hasProcessedPaymentIntent` is **not called** after a successful claim.

🟧 **Static vs instance call style.** `Stripe.webhooks.constructEvent` (static) replaces `stripe.webhooks.constructEvent` (instance). Both work, both verify signatures the same way. No bug, but the static form is unusual and may surprise SDK upgraders. Worth a comment.

🟩 **Idempotency-via-unique-index design.** `processed_stripe_events.event_id PRIMARY KEY` + `INSERT … ON CONFLICT DO NOTHING RETURNING` is the textbook idiom. Constant-time, atomic, race-proof — strict upgrade over the previous "scan recent N transactions" heuristic.

🟩 **Refund/dispute clawback** (lines 286–340 of the new file) is a real new capability. Allowing balance to go negative until top-up is the right tradeoff vs. blocking the clawback.

🟩 **Cleaner `paymentIntentId` extraction.** The old code did `const paymentIntentId = session.payment_intent as string;` — a runtime-unsafe cast that would coerce a Stripe `PaymentIntent` *object* (when expanded by the SDK) to the literal string `"[object Object]"`, silently corrupting the dedup metadata. The new code does `typeof session.payment_intent === "string" ? session.payment_intent : undefined`, which combined with the `if (!paymentIntentId) throw new Error(…)` guard fails loudly on the unexpected case. Side benefit of the refactor worth crediting.

🟨 **Refund-original lookup scans 500 recent rows** to find the matching purchase. Acknowledged in code. For a heavy-spending user this could miss an old purchase. Adding `creditTransactions.metadata->>'paymentIntentId'` as an indexed expression is the proper fix when scale demands.

🟨 **Currency check** in `routes/credits.ts:295,409` (`stripeCurrency !== EXPECTED_CURRENCY`) only fires when `stripeCurrency` is truthy. If the Stripe API ever returns a price with an unset currency, the check no-ops. Tighten to `if (stripeCurrency && stripeCurrency !== EXPECTED_CURRENCY)` → `if (!stripeCurrency || stripeCurrency !== EXPECTED_CURRENCY)`.

### `client/src/contexts/TransportContext.tsx`

🟨 **Async `play()` introduces stop-during-resume race.** Before:
```ts
const play = useCallback(() => {
  storePlay();
  const ctx = getAudioContext();
  if (ctx.state === 'suspended') ctx.resume();  // sync best-effort
  // schedule + start
}, [storePlay]);
```
After:
```ts
const play = useCallback(() => {
  void (async () => {
    await resumeAudioContext();
    storePlay();  // <-- moved inside async block
    const ctx = getAudioContext();
    // schedule + start
  })();
}, [storePlay]);
```
Scenario: user clicks Play, then Stop within ~10ms. `play()` returns synchronously while the `await resumeAudioContext()` is in flight. `stop()` runs and sets `isPlaying=false` + clears scheduler. Then the resume resolves, `storePlay()` flips it back to true, and `Tone.Transport.start('+0.05')` fires.

**Likely user-visible failure mode:** "I hit Stop and it kept playing for ~50–200ms then stopped." Annoying, but not data-corrupting — `Tone.Transport.start()` is internally idempotent on already-started state, and the scheduler cleanup in `stop()` would re-fire on the next user-initiated stop. Severity downgraded to 🟨 because no user data or audio corruption is at stake; only a brief UX hiccup. Worth fixing because the fix is small.

Fix: capture an epoch token before the await; check it after.
```ts
const epoch = ++playEpochRef.current;
await resumeAudioContext();
if (epoch !== playEpochRef.current) return;
storePlay();
…
```

🟨 **`storePlay()` moved inside the async block** changes the observable timing of `useStudioStore.isPlaying` for downstream consumers. UI components conditional on `isPlaying` will lag by the resume duration (often <1ms, but on cold iOS startup can be 100s of ms). If anything subscribes to `isPlaying` and immediately schedules audio against `Tone.Transport.position`, that's now mistimed.

Fix: replace the mock with a stateful in-memory store that simulates the table:
```ts
const events = new Map<string, { paymentIntentId?: string }>();
const paymentIntents = new Set<string>();
storage.tryClaimStripeEvent = vi.fn(async (e) => {
  if (events.has(e.eventId)) return false;
  events.set(e.eventId, { paymentIntentId: e.paymentIntentId });
  if (e.paymentIntentId) paymentIntents.add(e.paymentIntentId);
  return true;
});
storage.hasProcessedPaymentIntent = vi.fn(async (pi) => paymentIntents.has(pi));
```
With this fix, the existing "grants credits on first delivery" test would have caught the bug.

## Cross-cutting smells

1. **Three copies of `toggleStoryMode`** in `OrganismCommandCenter`, `OrganismControls`, `FloatingAudioMonitor` — extract to provider once.
2. **Defensive `typeof === 'function'` checks on typed context fields** — dead code that suggests someone wasn't sure of the contract. Trust the type system or fix the type.
3. **Magic numbers for "wait for engine to settle"** (`120ms`, `200ms`, `250ms` flush interval). Each is a ticking time bomb that breaks under cold start, slow CPUs, or load. Replace with deterministic ready signals where possible.
4. **Async-wrapped event handlers (`void (async () => …)()`) without epoch guards** — open the door to operation-during-await reverts. The play/stop case is the most visible; audit other call sites.
5. **String-literal escape leakage** (`\"x\"`, `'\''`, `&middot;` in JSX) suggests scripts/sed are part of the workflow. Add a final `prettier --check` or a manual diff scan to catch these.
6. **CLAUDE.md invariants now have ESLint enforcement for one rule (audio clock)** — extend the same pattern to the other invariants (e.g., `getAudioContext()` only, no direct credit math outside `creditService`).
